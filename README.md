# 引用视频理解

这是一个给 AstrBot 用的插件。

它的作用很简单：

- 用户平时发视频，插件只缓存，不会立刻让模型去“看视频”。
- 只有当用户“引用这条视频消息再提问”时，插件才会把视频内容真正注入给支持视频理解的模型。
- 这样做的好处是省 token，也更符合正常聊天习惯。

这个插件现在主要针对 QQ / OneBot v11 (`aiocqhttp`) 做了兼容优化。

## 这插件能解决什么问题

AstrBot 当前主线源码里，视频消息默认只会变成一段提示文字，比如：

```text
[Video Attachment: name demo.mp4, path D:\xxx\demo.mp4]
```

模型看到的只是“有一个视频文件路径”，并不是真的看到了视频内容。

这个插件会在合适的时候，把这段“文字提示”改造成真正的多模态视频输入，例如 `video_url`，这样支持视频理解的模型才有机会真的读视频。

## 适合谁用

如果你满足下面两个条件，就适合用这个插件：

1. 你在用 AstrBot。
2. 你接的是支持视频输入的模型，或者至少是 OpenAI 兼容的多模态接口。

比较适合的模型类型：

- Qwen / DashScope 视频模型
- OpenRouter 上支持视频的模型
- Kimi 新版视觉模型
- 其他明确支持 `video_url` 的 OpenAI 兼容接口

不适合的情况：

- 纯文本模型
- 不支持视频输入的普通图文模型
- 你希望“发视频就立刻自动分析”，那不是这个插件的设计目标

## 工作方式

插件分两步工作。

### 第一步：收到视频时，只缓存

用户在 QQ 里发一条视频消息时，插件会记住这条消息里有哪些视频，但不会调用模型。

这一步不花 token。

### 第二步：引用视频提问时，才注入视频

当用户这样做时：

```text
引用刚才那条视频
这段视频里发生了什么？
```

插件会尝试按下面顺序找到被引用的视频：

1. 先从 `Reply.chain` 里直接拿视频组件
2. 如果 QQ 适配器没把完整引用链带回来，就用 `Reply.id` 去插件缓存里找
3. 如果缓存也没命中，再从 AstrBot 默认生成的附件提示文本里把视频路径反解析出来

找到视频后，插件会根据当前模型类型，把视频转换成模型能吃的格式，然后注入到 LLM 请求里。

## QQ 为什么要做额外兼容

QQ / OneBot 的引用消息有时很完整，有时只给一个 `reply id`，不一定总能拿到原消息的完整内容。

这就是为什么这个插件专门加了两层 QQ 兜底：

- `Reply.id -> 插件缓存`
- 引用附件文本里的 `path -> 本地视频路径`

所以在 QQ 里，它比“只依赖 Reply.chain 的实现”稳很多。

## 安装位置

把插件目录放到 AstrBot 的插件目录里：

```text
AstrBot/data/plugins/astrbot_plugin_video_reference_vision
```

这个目录里至少应该有这些文件：

```text
main.py
metadata.yaml
_conf_schema.json
README.md
```

## 启用前先确认两件事

### 1. 你接的模型真的支持视频输入

如果模型本身不支持视频，这个插件也没法凭空让它支持。

### 2. AstrBot 运行目录下最好有 `data/config`

我已经顺手把 AstrBot 配置写入逻辑补强了。现在即使目录不存在，也会自动创建。

也就是说，插件首次加载时不需要你再手动补这个目录。

## 推荐配置

大多数人先用默认配置就够了。

如果你主要是 QQ + Qwen，可以先这样理解配置：

- `enabled`: 是否启用插件
- `mode`: 建议先用 `auto`
- `prefer_public_url`: 如果视频本来就是公网地址，优先直接传链接
- `max_base64_mb`: 本地视频转 base64 的大小上限，默认 20MB
- `qwen_fps`: Qwen 视频抽帧频率，默认 2.0
- `fallback_behavior`: 注入失败时怎么处理，建议先用 `keep_text`
- `allow_direct_video`: 默认关，表示“只有引用视频提问时才触发”

如果你用 Kimi：

- `kimi_strategy=auto`：默认自动判断
- `kimi_strategy=upload`：更偏向上传后再引用
- `kimi_upload_on_oversize=true`：视频太大时自动尝试上传，避免直接塞 base64

## 最常见的使用方式

### 场景一：正常推荐用法

1. 用户先发一条视频
2. 机器人此时不会分析
3. 用户引用这条视频，再问一句话
4. 插件注入视频
5. 模型回答视频内容

示例：

```text
用户先发：一条视频

用户再引用这条视频说：
这段视频里的人在做什么？
```

### 场景二：只发视频，不提问

插件只缓存，不调用模型。

这是正常行为，不是坏了。

### 场景三：引用了视频，但模型没回答出内容

常见原因：

- 当前模型不支持视频
- 视频太大，超出限制
- 提供商接口不接受本地 base64 视频
- QQ 那边引用链丢失，而且缓存也已经过期

## 失败时插件会怎么处理

插件默认不是“硬上”。

如果它判断当前模型不适合视频输入，或者视频处理失败，它会尽量不破坏 AstrBot 原本流程。

默认会回退到 AstrBot 原来的附件文本模式，也就是模型至少还能看到“有一个视频附件”这件事。

如果你把 `fallback_behavior` 设成 `silent`，那注入失败时它会更安静一些，不保留默认的视频附件提示文字。

## 目前已经做好的兼容点

- 引用触发，不乱耗 token
- QQ / OneBot v11 优先兼容
- `Reply.chain` 直接取视频
- `Reply.id -> 缓存` 取视频
- 从 AstrBot 默认的引用视频文本里反解析本地路径
- Qwen / DashScope 的 `video_url` 注入
- OpenRouter / 通用 OpenAI 兼容视频注入
- Kimi 的上传模式入口
- 成功注入后自动移除默认的 `[Video Attachment ...]` 文本，避免模型同时看到“真视频”和“本地路径提示”

## 目前没有做的事

下面这些不是当前版本的重点：

- 收到视频后立刻自动识别
- ffmpeg 抽帧兜底
- 音轨转写
- 所有厂商全量适配
- 把视频能力正式做进 AstrBot core

这是故意的，不是漏做。

第一版目标就是先让“引用视频提问”这条链路稳定跑通。

## 你可以怎么判断插件有没有正常工作

最简单的方法不是看代码，是看行为：

1. 在 QQ 里发一条视频
2. 再引用这条视频问一句明确的问题
3. 如果当前模型支持视频，它应该回答视频内容，而不是只重复“有一个视频附件”

如果它还是只回答“有个视频附件”，通常说明视频注入没有成功。

这时重点检查：

- 模型是否支持视频
- 插件是否启用
- 视频是否过大
- 日志里有没有“跳过处理”“视频过大”“Kimi 上传失败”之类的信息

## 一句总结

这个插件不是“自动分析所有视频”，而是“当你引用视频提问时，帮 AstrBot 把视频真正喂给支持视频理解的模型”。

如果你主要跑 QQ，这个插件的设计方向就是对的，而且已经把 QQ 最容易丢链路的几个坑补上了。
